# Publishing to npm

This guide walks through how to publish packages from this monorepo to npm.
It is written for someone who has never used Changesets before.

---

## What is a Changeset?

A **changeset** is a tiny text file (auto-generated by a CLI prompt) that answers two questions:

1. Which package(s) changed?
2. How big is the change — `patch`, `minor`, or `major`?

That's it. You write one whenever you finish a change that should eventually become a new npm release.
Changesets accumulate in the `.changeset/` folder until a maintainer decides it's time to publish.
Nothing gets published automatically — you always trigger it manually.

| Package | npm name | What it is |
| --- | --- | --- |
| `packages/gmt` | `@burglekitt/gmt` | Main Temporal library — updated frequently |
| `packages/gmt-biome` | `@burglekitt/gmt-biome` | Biome config — updated rarely |
| `packages/gmt-eslint` | `@burglekitt/gmt-eslint` | ESLint flat config — updated rarely |
| `packages/gmt-oxlint` | `@burglekitt/gmt-oxlint` | Oxlint JS plugin — updated rarely |

All four packages are published under the `@burglekitt` npm organization.
Each package has its own independent version number — bumping `gmt` to `0.3.0`
does not affect `gmt-eslint`.

## One-time setup (before your first publish)

### 1. npm org

The `@burglekitt` scope is an npm organization. To publish to it you must be a member.

- If the org does not exist yet: go to [npmjs.com/org/create](https://www.npmjs.com/org/create) and create it.
- If the org already exists: ask an existing owner to invite you.

### 2. npm access token

1. Log in at [npmjs.com](https://www.npmjs.com/).
2. Click your avatar → **Access Tokens** → **Generate New Token**.
3. Choose **Automation** (this bypasses 2FA and is safe for CI/CD).
4. Copy the token — you only see it once. It starts with `npm_`.

### 3. GitHub secret (for publishing via GitHub Actions)

1. Go to your repo on GitHub.
2. **Settings → Secrets and variables → Actions → New repository secret**.
3. Name: `NPM_TOKEN`, value: the token you just copied.
4. Go to **Settings → Environments** and create an environment named `npm-publish`.
5. Add `NPM_TOKEN` there as well (this gates the publish workflow to require approval if you configure it).

### 4. Local npm auth (for publishing from your machine)

If you publish locally, you authenticate via your existing npm login on your machine.
Run `npm whoami` to confirm you are logged in. If not, run `npm login` (or use `npm login --auth-type=web` for passkey/SSO).

## Day-to-day contributor flow

Every time you finish a change that will eventually ship as a release,
run this **on your feature branch before merging**:

```bash
bun run changeset:add
```

This is **fully interactive** — it does not auto-detect anything from git. It shows you a
checklist of all packages in the monorepo and you select which ones your change affected.

The CLI will:

1. **Show a list of all 4 packages** — use arrow keys to move, **spacebar to toggle selection**, Enter to confirm.
   You must know which packages your code change actually touched. If you only changed
   something in `packages/gmt`, select only `@burglekitt/gmt`.
2. **Ask for a bump type for each selected package:**
   - `patch` — bug fix, no API change (e.g. `0.2.0 → 0.2.1`)
   - `minor` — new feature, fully backwards-compatible (e.g. `0.2.0 → 0.3.0`)
   - `major` — breaking change, callers must update their code (e.g. `0.2.0 → 1.0.0`)
3. **Ask for a summary** — one sentence, written for someone reading the changelog.

This creates a small markdown file in `.changeset/` with a random name like `fluffy-cats-dance.md`.
Commit it alongside your code changes in the same PR.

> You do not have to run `changeset:add` for pure documentation changes,
> config tweaks, or test-only changes that do not affect the published package.

## Maintainer release flow (step-by-step)

This is what you do when you are ready to cut a new release.

### Step 1 — Check what is pending

```bash
bun run changeset status
```

This lists every package that has unreleased changesets and what version bump each one will get.
If it says "no changesets found", there is nothing to release yet.

### Step 2 — Apply versions and update changelogs

```bash
bun run changeset:version
```

This reads all the pending changeset files, bumps the `version` field in each affected `package.json`,
updates (or creates) the `CHANGELOG.md` for each affected package, and deletes the consumed
changeset files from `.changeset/`.

After this command, commit everything:

```bash
git add .
git commit -m "Version Packages"
git push
```

> Tip: The `release.yml` GitHub Actions workflow does Steps 1 and 2 automatically
> whenever changesets land on `main`. It opens a pull request called "Version Packages".
> Merging that PR is equivalent to running `changeset:version` and pushing.
> You can use either the automated PR or the manual command — they do the same thing.

### Step 3 — Build packages that need it

`gmt` and `gmt-oxlint` must be built before publishing. The publish workflow in GitHub Actions
handles this automatically. If you publish locally, build first:

```bash
# Build gmt
bunx nx run @burglekitt/gmt:build

# Build gmt-oxlint
cd packages/gmt-oxlint && bun run build && cd ../..
```

`gmt-biome` and `gmt-eslint` publish their source files as-is — no build needed.

### Step 4 — Dry run (recommended)

Verify what will be included in each package before uploading anything:

```bash
cd packages/gmt       && npm pack --dry-run && cd ../..
cd packages/gmt-biome && npm pack --dry-run && cd ../..
cd packages/gmt-eslint && npm pack --dry-run && cd ../..
cd packages/gmt-oxlint && npm pack --dry-run && cd ../..
```

Check that no unexpected files are included and that the listed version number is correct.

### Step 5 — Publish

There are two paths. Use whichever fits your situation.

#### Path A — GitHub Actions (recommended)

The safest option. Handles authentication, build, and publish with full logging.

1. Go to the repo on GitHub.
2. Click **Actions** → **Publish Package**.
3. Click **Run workflow** (top right).
4. Choose which package to publish and which npm tag (`latest`, `next`, `beta`).
5. Click the green **Run workflow** button.

Note: this workflow uses `npm publish` directly. It does **not** create git tags automatically.
After the workflow succeeds, do Step 6 below to tag.

#### Path B — Local publish

Requires being logged in to npm on your machine (`npm whoami` to verify).

```bash
bun run changeset:publish
```

This runs `changeset publish`, which:

1. Checks which packages have a version in `package.json` that hasn't been published to npm yet.
2. Runs `npm publish --access public` for each unpublished package.
3. **Automatically creates a git tag** for each successful publish (e.g. `@burglekitt/gmt@0.3.0`).

Then push the tags (Step 6).

### Step 6 — Create and push git tags

#### After Path A (GitHub Actions publish)

The Actions workflow does not create tags. After it finishes:

```bash
# Pull the latest commits from main (the version bump from "Version Packages" PR)
git pull

# Create git tags for any package whose current version does not have a tag yet
bunx changeset tag

# Push the new tags to GitHub
git push --follow-tags
```

#### After Path B (local publish)

`bun run changeset:publish` already created the tags. Just push them:

```bash
git push --follow-tags
```

## Git tags in this monorepo

### Why monorepo tags look different

In a single-package repo, a tag like `v1.2.3` refers to the whole project.
In a monorepo with multiple independently-versioned packages, that format is ambiguous —
`v1.2.3` of *which* package?

Changesets solves this by scoping each tag to the package it belongs to:

| Package | Example tag |
| --- | --- |
| `@burglekitt/gmt` | `@burglekitt/gmt@0.3.0` |
| `@burglekitt/gmt-biome` | `@burglekitt/gmt-biome@0.1.2` |
| `@burglekitt/gmt-eslint` | `@burglekitt/gmt-eslint@0.1.2` |
| `@burglekitt/gmt-oxlint` | `@burglekitt/gmt-oxlint@0.2.0` |

Each package gets its own tag, independently. If you release `gmt` three times without
touching `gmt-biome`, there will be three `@burglekitt/gmt@x.y.z` tags and zero new
`@burglekitt/gmt-biome@x.y.z` tags.

### How tags are created

When `changeset publish` runs (either locally via `changeset:publish` or inside
the `publish.yml` workflow), it creates a git tag for each package it successfully publishes.

```text
@burglekitt/gmt@0.3.0
@burglekitt/gmt-oxlint@0.2.0
```

Push them to the remote:

```bash
git push --follow-tags
```

### Viewing tags

```bash
# List all changeset-style tags
git tag --list '@burglekitt/*'

# See all tags sorted by version
git tag --sort=version:refname
```

## Semantic versioning cheat-sheet

You do **not** run different commands for different bump types. The bump type is always
chosen interactively when you run `bun run changeset:add` — the CLI will ask
"Is this a patch, minor, or major bump?" and you pick with arrow keys.

| Change type | Pick in CLI | Version example |
| --- | --- | --- |
| Internal refactor, no behavior change | skip — no changeset needed | — |
| Bug fix | `patch` | `0.2.0 → 0.2.1` |
| New optional parameter or export | `minor` | `0.2.1 → 0.3.0` |
| Renamed export, removed export, changed return type | `major` | `0.3.0 → 1.0.0` |

## Quick reference

```bash
# --- On your feature branch, before merging the PR ---

# Interactive prompt: pick which packages changed and what kind of bump
bun run changeset:add
git add .changeset/
git commit -m "add changeset"

# --- When ready to release (on main, after PRs are merged) ---

# See what versions would be bumped today
bun run changeset status

# Bump versions in package.json files and regenerate CHANGELOGs
# (or merge the auto-opened "Version Packages" PR instead)
bun run changeset:version
git add .
git commit -m "Version Packages"
git push

# --- Publishing ---

# Option A: GitHub Actions (recommended) — go to Actions → Publish Package → Run workflow
# After it finishes, create and push tags:
git pull
bunx changeset tag
git push --follow-tags

# Option B: Local publish (must be logged in to npm — run `npm whoami` to check)
bun run changeset:publish   # publishes + auto-creates tags
git push --follow-tags
```
